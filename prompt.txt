根据您提供的协议片段，以下是针对“帧结构（Big-Endian）”的详细整理与架构实现逻辑解析。

### 1. 帧结构定义表

该协议采用标准的大端（Big-Endian）序列，最小帧长为 4 字节，最大为 63 字节。

| Byte 索引 | 名称 | 固定值/定义 | 说明 |
| --- | --- | --- | --- |
| 0 | Header | `0x55` | 帧头起始 |
| 1 | Header | `0xAA` | 帧头标识 |
| 2 | CMD_ID | `0xDC` | 命令/控制标识（在此场景下固定） |
| 3 | Body Length | `n` | **关键位域解析** (见下文) |
| 4 ~ n+1 | Data | 可变 | 具体指令参数（如角度、开关状态） |
| n+2 | CHECKSUM | `CS` | 异或校验和 |

---

### 2. 核心字段深度解析

#### 2.1 长度字段 (Byte 3) 的位域映射

该字节不仅记录长度，还包含了帧计数器，在编码实现时必须使用 **位运算 (Bitwise Operation)** 拆解：

* **bit 0 ~ 5 (低 6 位)**：表示 `Body Length (n)`。
* `n` 的定义：从 `Byte 3` 到 `Checksum` 位的所有字节总和（不含校验位本身）。
* 最小 `n=4`，最大 `n=63`。


* **bit 6 ~ 7 (高 2 位)**：表示 `Frame Counter`（帧计数器）。
* 该值应在每次发送新帧时递增（范围 0-3），用于接收方判断丢包或重复。



#### 2.2 校验和 (CHECKSUM) 计算规则

该协议要求对帧内的数据进行异或运算，这对于保障串口通信（UART）的完整性至关重要。

* **计算范围**：从 `Byte 3` (包含长度位) 到 `Byte n+1` (数据包最后一个字节)。
* **逻辑算法**：
```python
# 伪代码逻辑
checksum = 0
for i in range(3, n + 2): # 从字节3开始到数据末尾
    checksum ^= buffer[i]

```

### 3. 工程实现建议（Codex 友好）

在开发探照灯管理器时，建议封装一个 `ProtocolPacker` 类来处理这些逻辑，以防止手动拼包出错。

#### 伪代码实现示例

```python
def create_frame(cmd_id, data, frame_counter=0):
    # 1. 组装基础部分
    body = [cmd_id] + data
    n = len(body) # 计算长度
    
    # 2. 构造 Byte 3: 长度 + 计数器
    # (n & 0x3F) 确保长度不超过6位，(counter << 6) 将计数器移至高位
    len_byte = (n & 0x3F) | ((frame_counter & 0x03) << 6)
    
    # 3. 构造完整帧并计算 Checksum
    full_frame = [0x55, 0xAA, 0xDC, len_byte] + data
    checksum = 0
    for byte in full_frame[3:]:
        checksum ^= byte
    
    return full_frame + [checksum]

```
### 4. 架构提示

* **大端序（Big-Endian）约束**：如果您的控制系统运行在 ARM 或 x86 (小端) 环境中，涉及多字节数值（如 16 位角度值 `0x1554`）时，**必须显式进行字节交换**（例如：`15 54`，高位在前）。
* **校验一致性**：在实现串口读取时，必须先读取 `Byte 3` 确定帧长 `n`，再按 `n+1` 个字节读取剩余数据，最后对比 `Checksum`。若校验失败，必须清空接收缓冲区，防止帧错位。
